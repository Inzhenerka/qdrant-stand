# Основы SQL и реляционных БД

## Зачем этот курс

Большая часть прикладных данных хранится не в CSV-файлах, а в реляционных базах данных: PostgreSQL, MySQL, SQLite, Oracle, SQL Server. SQL — общий язык запросов ко всем этим системам: освоив его один раз, вы умеете доставать и менять данные практически в любой компании. Требование «SQL — обязательно» стандартно для аналитиков, бэкендеров, дата-инженеров и тестировщиков.

К концу модуля вы научитесь:

- выбирать и фильтровать строки через `SELECT … FROM … WHERE`,
- сортировать и брать уникальные значения через `ORDER BY` / `DISTINCT`,
- вычислять новые значения прямо в `SELECT` и давать им псевдонимы,
- правильно работать с `NULL` через `IS NULL` / `IS NOT NULL`,
- считать агрегаты (`count`, `sum`, `avg`, `min`, `max`),
- группировать через `GROUP BY` и фильтровать группы через `HAVING`,
- связывать таблицы через `INNER JOIN` / `LEFT JOIN`,
- изменять данные через `INSERT` / `UPDATE` / `DELETE`,
- создавать таблицы с ограничениями целостности,
- ускорять запросы индексами и атомарно изменять данные транзакциями,
- работать с SQL из Python через `sqlite3`, безопасно от SQL-инъекций.

Предполагается знание базового Python (модуль «Введение в Python») для последнего эпизода. До него SQL осваивается без кода вообще.

## Подготовка к работе

Курс построен на **SQLite** — самой простой и распространённой реляционной БД. Главная особенность: вся база лежит в **одном файле** на диске. Никакого отдельного сервера, никакой настройки прав — открыл файл, выполнил запрос, закрыл. Идеально для учёбы и многих продакшен-сценариев.

**Установка:**

- **Windows:** скачайте `sqlite-tools` с sqlite.org, распакуйте, добавьте в PATH. Альтернатива — поставить SQLite через Git Bash или WSL.
- **macOS:** SQLite уже стоит. Проверьте: `sqlite3 --version`.
- **Linux:** `sudo apt install sqlite3` (Debian/Ubuntu).

**Клиенты:**

- **`sqlite3` CLI** — встроенный консольный клиент, в нём идут все примеры этого модуля.
- **DB Browser for SQLite** — бесплатный графический клиент, удобен для рассматривания таблиц.
- **DBeaver** — универсальный GUI, поддерживает SQLite, PostgreSQL, MySQL и десятки других СУБД. Пригодится дальше курса.

**Учебная БД.** В рамках курса используется файл `survey.db` со сценарием антарктической экспедиции (см. ниже). Скачать его можно с страницы курса; чтобы открыть:

```bash
sqlite3 survey.db
sqlite> .tables                    # список таблиц
sqlite> .schema Person              # структура таблицы Person
sqlite> SELECT * FROM Person;
sqlite> .quit
```

## Сценарий курса

Чтобы примеры были осмысленными, мы используем датасет научной экспедиции в Антарктику первой половины XX века — классический учебный набор. В нём есть всё, ради чего реляционная модель и нужна: пропуски, аномалии и связи между таблицами.

Данные распределены по четырём связанным таблицам:

- **`Person`** — список исследователей. Столбцы: `id`, `personal` (имя), `family` (фамилия). Например: `dyer`, `lake`, `roe`, `pb`.
- **`Site`** — точки сбора измерений. Столбцы: `name` (например, `DR-1`, `DR-3`, `MSK-4`), `lat`, `long`.
- **`Visited`** — посещения сайтов. Столбцы: `id` (идентификатор визита), `site`, `dated`.
- **`Survey`** — собственно измерения: `taken` (id визита), `person` (кто измерял), `quant` (тип величины: `rad` — радиация, `sal` — солёность, `temp` — температура), `reading` (значение).

## Эпизод 1. SELECT и FROM — выборка данных

Самая частая операция SQL — выборка строк из таблицы:

```sql
SELECT * FROM Person;
```

`*` означает «все столбцы». Если нужны конкретные:

```sql
SELECT family, personal FROM Person;
```

Несколько важных особенностей синтаксиса:

- SQL **регистронезависим** для ключевых слов и имён: `SELECT`, `select`, `Select` — одно и то же. Принято писать ключевые слова заглавными буквами для читаемости.
- Сами **данные** регистрозависимы: `'Lake'` и `'lake'` — разные строки.
- Каждая команда заканчивается точкой с запятой `;`.
- **Порядок строк** в результате не гарантирован, если не задана сортировка.

**Комментарии в SQL:**

```sql
-- Однострочный комментарий до конца строки
SELECT * FROM Person;   -- можно и в конце строки

/* Многострочный
   комментарий */
```

В сложных запросах с подзапросами и JOIN'ами комментарии экономят часы, когда к запросу возвращаешься или передаёшь его другому.

## Эпизод 2. ORDER BY и DISTINCT

Сортировка результата:

```sql
SELECT * FROM Person ORDER BY family;             -- по возрастанию (ASC — по умолчанию)
SELECT * FROM Site ORDER BY lat DESC;             -- по убыванию
SELECT * FROM Survey ORDER BY taken, quant;       -- по двум столбцам
SELECT * FROM Survey ORDER BY taken ASC, quant DESC;
```

`ASC` (ascending) — по возрастанию, это поведение по умолчанию, явно писать не обязательно. `DESC` (descending) — по убыванию.

Уникальные значения столбца:

```sql
SELECT DISTINCT quant FROM Survey;       -- какие типы измерений вообще есть
SELECT DISTINCT site, dated FROM Visited;
```

`DISTINCT` смотрит на **комбинацию** перечисленных столбцов, а не на каждый по отдельности.

## Эпизод 3. WHERE — фильтрация

`WHERE` отбирает строки по условиям:

```sql
SELECT * FROM Visited WHERE site = 'DR-1';
SELECT * FROM Visited WHERE site = 'DR-1' AND dated < '1930-01-01';
SELECT * FROM Survey WHERE quant = 'sal' AND person = 'lake';
```

Логические операторы — `AND`, `OR`, `NOT`. При сложных условиях используйте круглые скобки, чтобы избежать неоднозначности приоритетов.

Принадлежность списку — `IN`:

```sql
SELECT * FROM Survey WHERE person IN ('lake', 'roe');
```

Шаблоны строк — `LIKE` с подстановочными символами:

- `%` — любое число любых символов (включая ноль)
- `_` — ровно один любой символ

```sql
SELECT * FROM Visited WHERE site LIKE 'DR%';      -- все сайты, начинающиеся на DR
SELECT * FROM Visited WHERE site LIKE 'DR-_';     -- DR-1, DR-3, но не DR-12
SELECT * FROM Person WHERE family LIKE '%son';    -- заканчивается на son
```

Диапазон — `BETWEEN`:

```sql
SELECT * FROM Survey WHERE reading BETWEEN 0.0 AND 1.0;
```

`BETWEEN` включает обе границы.

**Важно.** `WHERE` работает построчно по исходным данным. В нём можно использовать столбцы, которых нет в `SELECT` — например, отбирать по дате, а возвращать только сайт.

## Эпизод 4. Вычисления в SELECT

В `SELECT` можно не только выбирать, но и вычислять новые значения:

```sql
SELECT 1.05 * reading FROM Survey WHERE quant = 'rad';
SELECT taken, round(reading, 2) FROM Survey WHERE quant = 'sal';
SELECT personal || ' ' || family FROM Person;     -- конкатенация строк
```

Псевдоним столбца — через `AS`:

```sql
SELECT 1.05 * reading AS adjusted_radiation FROM Survey WHERE quant = 'rad';
```

Псевдонимы делают вывод читаемым и позволяют ссылаться на вычисление в `ORDER BY` / `HAVING`.

**Замечание о диалектах.** SQL — это семейство языков; разные СУБД отличаются в деталях. Например, конкатенация строк:

- **SQLite, PostgreSQL, Oracle:** `'Hello' || ' ' || 'world'`
- **MySQL:** `CONCAT('Hello', ' ', 'world')`
- **SQL Server:** `'Hello' + ' ' + 'world'` или `CONCAT(...)`.

Базовые `SELECT`/`WHERE`/`JOIN` работают одинаково везде, но при переходе на новую СУБД проверяйте функции и спецсинтаксис в её документации.

## Эпизод 5. Пропущенные значения (NULL)

В реальных данных часть значений отсутствует. SQL обозначает это специальным значением `NULL`. Главное про `NULL`: это «неопределённое значение», а не «пусто», и любая операция с ним даёт снова `NULL`.

```sql
SELECT * FROM Visited WHERE dated < '1930-01-01';
-- Строки, где dated равно NULL, не попадут — даже если они «явно раньше».
```

Проверять `NULL` нужно специальными операторами `IS NULL` и `IS NOT NULL`, а не `=`:

```sql
SELECT * FROM Visited WHERE dated IS NULL;
SELECT * FROM Visited WHERE dated IS NOT NULL;
```

Это одна из самых частых причин неправильных запросов у новичков: написал `WHERE x != 5` и удивился, почему строки с `NULL` тоже не попадают в результат. Они и не должны — `NULL != 5` — это не `TRUE`, это `NULL`.

Чтобы заменить `NULL` на конкретное значение в выводе — функция `COALESCE`:

```sql
SELECT person, COALESCE(reading, 0) FROM Survey;   -- NULL заменить на 0
```

## Эпизод 6. Агрегация

Агрегатные функции принимают набор строк и возвращают одно значение:

```sql
SELECT min(dated) FROM Visited;                              -- 1927-02-08
SELECT max(dated) FROM Visited;                              -- 1932-03-22
SELECT count(*) FROM Survey;                                 -- общее количество
SELECT count(reading) FROM Survey WHERE quant = 'sal';       -- 9
SELECT sum(reading) FROM Survey WHERE quant = 'sal';         -- 64.83
SELECT avg(reading) FROM Survey WHERE quant = 'sal';         -- 7.20
```

Агрегатные функции игнорируют `NULL` — это удобно для неполных данных.

**Тонкость:** `count(*)` считает все строки, `count(reading)` — только те, где `reading` не `NULL`. Разница важна при анализе пропусков.

## Эпизод 7. GROUP BY и HAVING

Часто агрегат нужен **по группам**:

```sql
SELECT person, count(reading), round(avg(reading), 2)
FROM Survey
WHERE quant = 'rad'
GROUP BY person;
```

Этот запрос возвращает по одной строке на каждого исследователя с количеством его измерений радиации и средним значением. Группировка возможна и по нескольким полям:

```sql
SELECT person, quant, count(reading)
FROM Survey
GROUP BY person, quant;
```

**Тонкий момент.** `WHERE` применяется к строкам **до** группировки, а `HAVING` — к группам **после**. Если нужно отфильтровать по агрегату — это `HAVING`, не `WHERE`:

```sql
SELECT person, avg(reading) AS mean_rad
FROM Survey
WHERE quant = 'rad'
GROUP BY person
HAVING mean_rad > 5.0;
```

Запомните порядок: `FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY`. Так читается ход исполнения, хотя пишут разделы в другом порядке.

## Эпизод 8. JOIN — связывание таблиц

Реляционная модель распределяет данные по нескольким таблицам, чтобы избежать дублирования. Чтобы получить связанную картинку, таблицы соединяют через `JOIN`.

`JOIN` сам по себе создаёт **декартово произведение** — каждая строка одной таблицы комбинируется с каждой строкой другой. Это редко то, что нужно. Условие связывания задаётся через `ON`:

```sql
SELECT Site.lat, Site.long, Visited.dated
FROM Site
JOIN Visited ON Site.name = Visited.site;
```

Так мы соединяем сайты с их посещениями: для каждого визита получаем координаты сайта.

Несколько JOIN можно цеплять подряд. Получим тип измерения и значение вместе с координатами сайта и датой:

```sql
SELECT Site.lat, Site.long, Visited.dated, Survey.quant, Survey.reading
FROM Site
JOIN Visited ON Site.name = Visited.site
JOIN Survey ON Visited.id = Survey.taken;
```

### Алиасы таблиц

В реальных запросах имена таблиц длинные, а ссылок на них — много. Поэтому таблицам дают короткие псевдонимы через `AS` (или просто пробел):

```sql
SELECT s.lat, s.long, v.dated, sv.quant, sv.reading
FROM Site AS s
JOIN Visited AS v ON s.name = v.site
JOIN Survey AS sv ON v.id = sv.taken;
```

Запрос становится компактнее и читабельнее. Это стандартный приём, в больших проектах вы будете видеть его повсюду.

### Ключи

Чтобы такие связи работали корректно, в каждой таблице должны быть:

- **Первичный ключ (primary key)** — поле, уникально идентифицирующее строку. Например, `Person.id`, `Site.name`, `Visited.id`.
- **Внешний ключ (foreign key)** — поле, ссылающееся на первичный ключ другой таблицы. Например, `Survey.person` ссылается на `Person.id`, `Survey.taken` ссылается на `Visited.id`, `Visited.site` ссылается на `Site.name`.

БД сама проверяет, что внешние ключи указывают на существующие записи — это гарантирует целостность данных. (В SQLite проверку нужно включить вручную: `PRAGMA foreign_keys = ON;`.)

### Типы JOIN

- **`INNER JOIN`** (или просто `JOIN`) — только строки, у которых есть пара в обеих таблицах.
- **`LEFT JOIN`** — все строки из левой таблицы; для не имеющих пары в правой — `NULL` справа.
- **`RIGHT JOIN`** — наоборот.
- **`FULL OUTER JOIN`** — все строки из обеих, с `NULL`-ами там, где пары нет.

**Совместимость.** `RIGHT JOIN` и `FULL OUTER JOIN` поддерживаются не во всех СУБД (в SQLite появились только в версии 3.39, в 2022 году; в более старых их нет). Полезное правило: любой `RIGHT JOIN A ON ...` эквивалентен `LEFT JOIN A ON ...` после перемены местами таблиц, а это работает везде. Поэтому в реальных запросах почти всегда используют `INNER JOIN` и `LEFT JOIN` — этой пары хватает на 99% случаев.

## Эпизод 9. Создание и изменение данных

SQL умеет не только читать, но и менять данные:

```sql
INSERT INTO Person (id, personal, family)
VALUES ('skol', 'Skywalker', 'Lars');

UPDATE Site SET lat = -47.55 WHERE name = 'DR-1';

DELETE FROM Person WHERE id = 'skol';
```

**Важно.** `UPDATE` и `DELETE` без `WHERE` тронут **все** строки таблицы. Перед запуском на боевой БД сначала прогоните то же `WHERE` через `SELECT` и убедитесь, что отбираются именно те строки. И оборачивайте такие операции в транзакции (см. эпизод 11).

## Эпизод 10. Создание таблиц

```sql
CREATE TABLE Person (
    id TEXT PRIMARY KEY,
    personal TEXT NOT NULL,
    family TEXT NOT NULL
);

CREATE TABLE Survey (
    taken INTEGER NOT NULL,
    person TEXT,
    quant TEXT NOT NULL,
    reading REAL,
    FOREIGN KEY (taken) REFERENCES Visited(id),
    FOREIGN KEY (person) REFERENCES Person(id)
);
```

Ключевые модификаторы:

- `PRIMARY KEY` — первичный ключ.
- `NOT NULL` — поле обязательно для заполнения.
- `UNIQUE` — значения не могут повторяться.
- `DEFAULT <значение>` — значение по умолчанию.
- `CHECK (условие)` — ограничение целостности (например, `CHECK (reading >= 0)`).
- `FOREIGN KEY ... REFERENCES ...` — внешний ключ.

Хорошая привычка — задавать ограничения целостности **на уровне БД**, а не только в коде приложения. Тогда даже если кто-то полезет менять данные через консоль, БД не даст нарушить инварианты.

## Эпизод 11. Индексы и транзакции

Если по столбцу часто идёт фильтрация или сортировка — заведите **индекс**:

```sql
CREATE INDEX idx_survey_quant ON Survey(quant);
```

Индексы ускоряют `SELECT`, но замедляют `INSERT`/`UPDATE` и занимают место. Первичные ключи и `UNIQUE`-поля индексируются автоматически. Не создавайте индексы «на всякий случай» — добавляйте, когда конкретный запрос реально тормозит.

**Транзакции** — способ объединить несколько изменений как одно атомарное действие («всё или ничего»):

```sql
BEGIN TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;            -- зафиксировать
-- или: ROLLBACK; -- откатить, как будто ничего не было
```

Классический пример — перевод денег: если между двумя `UPDATE` отключится свет, без транзакции деньги уйдут с одного счёта, но не придут на другой. С транзакцией БД либо применит оба изменения, либо ни одного.

## Эпизод 12. SQL из Python

Стандартная библиотека Python содержит модуль `sqlite3` — встроенную БД SQLite, которая хранит данные в одном файле без отдельного сервера:

```python
import sqlite3

conn = sqlite3.connect("survey.db")
cursor = conn.cursor()

cursor.execute("SELECT * FROM Person WHERE id = ?", ("dyer",))
for row in cursor.fetchall():
    print(row)

conn.close()
```

### Современный паттерн через with

`sqlite3.connect` поддерживает контекстный менеджер — `conn` сам закроется по выходу из блока, что особенно важно при ошибках посередине:

```python
with sqlite3.connect("survey.db") as conn:
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM Person WHERE id = ?", ("dyer",))
    print(cursor.fetchall())
```

### conn.commit() для изменений

При `INSERT`, `UPDATE`, `DELETE` нужно явно зафиксировать транзакцию:

```python
with sqlite3.connect("survey.db") as conn:
    conn.execute("INSERT INTO Person VALUES (?, ?, ?)", ("skol", "Skywalker", "Lars"))
    conn.commit()                # без этого строки не сохранятся в файл!
```

Это **классическая ловушка новичка**: «код выполнился, ошибок нет, а в БД пусто». Причина — забыли `commit`. SQLite в режиме `with conn:` коммитит автоматически по выходу из блока, но в явном коде без `with` нужно вызвать самому.

### Безопасность: параметризация

Никогда не подставляйте значения в запрос через f-string или конкатенацию строк — это уязвимость SQL-инъекций:

```python
# ПЛОХО: уязвимость SQL-инъекции
user_input = "dyer'; DROP TABLE Person; --"
cursor.execute(f"SELECT * FROM Person WHERE id = '{user_input}'")   # катастрофа

# ХОРОШО: параметризация
cursor.execute("SELECT * FROM Person WHERE id = ?", (user_input,))   # безопасно
```

В первом случае пользовательский ввод подставляется как часть SQL — и злоумышленник может выполнить любой код. Во втором БД сама экранирует значение, что бы в нём ни было.

Параметры в `sqlite3` — `?`; в драйверах для PostgreSQL/MySQL обычно `%s`. Всегда используйте этот механизм для пользовательских данных.

### ORM в больших проектах

Для проектов на десятки таблиц вместо ручных SQL применяют ORM-библиотеки — **SQLAlchemy**, **Django ORM**. Они генерируют SQL из объектных моделей, делают код переносимым между разными СУБД и автоматизируют миграции схемы. Знание базового SQL никуда не девается — без него отладка проблем ORM превращается в гадание.
