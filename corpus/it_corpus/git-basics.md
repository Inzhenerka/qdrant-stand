# Основы Git

## Эпизод 1. Зачем нужен автоматический контроль версий

Представьте, что вы пишете большой документ — методическое пособие, статью, дипломную работу. После каждой правки вы сохраняете новую копию, и через месяц рядом лежат файлы вроде `posobie_v3_FINAL_правки_иванова_FINAL2.docx`. Какая из них самая свежая? Какая правка от какого числа? Что было в той версии, которую вы согласовали с коллегой неделю назад?

Системы контроля версий (Version Control Systems, VCS) решают именно эту проблему. По смыслу это **«неограниченная отмена»**: вы можете вернуться к любой предыдущей версии файла, посмотреть, что и когда было изменено, и кто это менял. Файл при этом остаётся один — все версии хранятся в специальной служебной структуре.

Современные VCS работают по такой модели: исходный файл и **набор изменений** хранятся отдельно. Систему можно «попросить» проиграть нужную последовательность изменений на исходник — и получить любую промежуточную версию.

Такая модель естественно решает и **совместную работу**. Двое могут одновременно редактировать разные участки одного файла; система склеит их правки автоматически. Если же оба изменили одну и ту же строку по-разному — возникает **конфликт**, и система сообщит об этом, не давая никому случайно затереть чужую работу.

Git — самая распространённая VCS в мире. Это **распределённая** система: каждый участник имеет полную копию истории на своём компьютере, центральный сервер не обязателен. На практике центральный сервер всё же используется (GitHub, GitLab) — он удобен как точка обмена и резервная копия.

## Эпизод 2. Настройка Git

Перед первым использованием Git нужно представиться: указать имя и e-mail, которые будут привязаны к каждому коммиту:

```bash
git config --global user.name "Анна Петровна"
git config --global user.email "anna@example.com"
```

Флаг `--global` означает «для всех репозиториев на этом компьютере». Без него настройки применяются только к текущему репозиторию.

E-mail должен совпадать с тем, что указан в вашей учётной записи на GitHub (или другом хостинге) — иначе ваши коммиты не будут связываться с вашим профилем.

**Окончания строк.** На Windows строки исторически заканчиваются `\r\n`, на Linux и macOS — `\n`. Чтобы Git не ломал их при переносе кода между системами, настраивают `core.autocrlf`:

```bash
git config --global core.autocrlf input    # Linux / macOS
git config --global core.autocrlf true     # Windows
```

**Текстовый редактор для сообщений коммитов.** По умолчанию Git открывает Vim, что для новичка может стать неприятным сюрпризом. Можно выбрать привычный редактор:

```bash
git config --global core.editor "nano -w"
git config --global core.editor "code --wait"   # VS Code
```

**Имя главной ветки.** Раньше по умолчанию называлась `master`, сейчас принято `main`:

```bash
git config --global init.defaultBranch main
```

Эти команды выполняются **один раз на компьютер**. Посмотреть текущие настройки:

```bash
git config --list --global
git config --global --edit       # открыть в редакторе
```

## Эпизод 3. Создаём репозиторий

Дальше курс будет идти по сквозному примеру: шеф Alfredo Linguini собирает свою цифровую кулинарную книгу. Он работает в папке `recipes/` и хочет отслеживать каждую правку каждого рецепта.

Создаём папку и переходим в неё:

```bash
mkdir recipes
cd recipes
```

Превращаем её в Git-репозиторий:

```bash
git init
```

Эта команда создаёт скрытую служебную папку `.git`, в которой Git будет хранить всю историю проекта. Удалите эту папку — и история исчезнет.

Проверим, что репозиторий пуст и готов к работе:

```bash
git status
```

Git ответит, что коммитов ещё нет и нет файлов для отслеживания.

**Важное предупреждение.** Никогда не создавайте Git-репозиторий внутри другого Git-репозитория. Внешний попытается отслеживать содержимое внутреннего, и оба будут друг другу мешать. Если случайно так получилось — удалите вложенную папку `.git` (только её, не данные!).

## Эпизод 4. Отслеживаем изменения

Alfredo создаёт первый рецепт — файл `guacamole.md`. Положим в него заголовок и небольшой текст:

```markdown
# Гуакамоле

Простая мексиканская закуска из авокадо.
```

Сохраняем, проверяем статус:

```bash
git status
```

Git покажет файл `guacamole.md` как **untracked** — он есть в папке, но Git его пока не отслеживает. Чтобы файл попал в историю, нужны два шага:

1. **`git add`** — положить изменения в так называемую *staging area* (промежуточную область, индекс).
2. **`git commit`** — зафиксировать содержимое индекса как новую версию.

Полезная аналогия: представьте, что вы делаете групповое фото. `git add` — это собрать нужных людей перед камерой. `git commit` — нажать кнопку и сделать снимок. Можно собирать группу постепенно (несколько `git add` подряд), а потом одним коммитом зафиксировать всех вместе.

```bash
git add guacamole.md
git status              # теперь файл "to be committed"
git commit -m "Добавить рецепт гуакамоле"
```

Флаг `-m` задаёт сообщение коммита прямо в команде. Без него Git откроет настроенный текстовый редактор.

После коммита `git status` покажет «working tree clean» — все изменения зафиксированы.

**Правила хорошего сообщения коммита:**

- Первая строка — короткая (до 50 символов), в повелительном наклонении: «Добавить…», «Исправить…», «Обновить…».
- При необходимости — пустая строка и подробности ниже.
- Описывайте **зачем** сделано изменение, не только **что**. Через год вы себе спасибо скажете.

Теперь Alfredo вносит правку в `guacamole.md` — добавляет ингредиенты. Прежде чем коммитить, он хочет убедиться, что именно изменилось:

```bash
git diff
```

`git diff` показывает разницу между рабочей копией и последней зафиксированной версией. Строки, помеченные `+`, добавлены; помеченные `-` — удалены.

Цикл повторяется: правка → `git add` → (опц. `git diff --staged`) → `git commit -m "…"`.

## Эпизод 5. Изучаем историю

Просмотр всех коммитов:

```bash
git log
git log --oneline           # короткая форма, по строке на коммит
```

Каждый коммит идентифицируется **хешем** — уникальной строкой из 40 символов в шестнадцатеричной форме, например `f22b25e3233b4645dabd0d81e651fe074bd8e73b`. На практике достаточно первых 7 символов: `f22b25e`.

Также есть относительные ссылки:
- `HEAD` — самый последний коммит в текущей ветке.
- `HEAD~1` — предпоследний.
- `HEAD~2` — на два коммита назад. И так далее.

Сравнить рабочую копию с разными состояниями:

```bash
git diff HEAD               # с последним коммитом
git diff HEAD~1             # с предпоследним
git diff f22b25e            # с конкретным коммитом по хешу
git show HEAD~2             # текст и сообщение коммита
```

**Откат изменений.** Если правка в рабочей копии оказалась неудачной, её можно отменить:

```bash
git restore guacamole.md             # вернуть последнюю закоммиченную версию
git restore -s f22b25e guacamole.md  # вернуть состояние из конкретного коммита
git restore --staged guacamole.md    # снять файл со staging area
```

Если файл уже был добавлен в индекс (`git add`), но ещё не закоммичен, и вы хотите отменить и индексирование, и сами правки — два шага: сначала `git restore --staged`, потом `git restore`.

**Полезный приём.** Разделяйте файлы по логическим частям (введение отдельно, заключение отдельно). Это резко упрощает откат: вы вернёте только то, что испортили, не трогая остальное.

## Эпизод 6. Игнорируем лишнее

В проектах часто появляются файлы, которые отслеживать не нужно: временные файлы редактора, кэш, скомпилированные версии, личные заметки, секреты. Чтобы Git перестал на них реагировать, создайте в корне репозитория файл `.gitignore` и перечислите шаблоны:

```
*.tmp
results/
**/*.csv
.env
notes_личное.md
```

Что означают эти строки:

- `*.tmp` — все файлы с расширением `.tmp` в любой папке.
- `results/` — вся папка `results` со всем содержимым.
- `**/*.csv` — все `.csv`-файлы на любом уровне вложенности.
- `.env` — конкретный файл `.env`.

Восклицательный знак отменяет игнорирование для конкретного файла:

```
*.png
!logo.png      # все PNG игнорируются, но logo.png отслеживается
```

После создания `.gitignore` команда `git status` перестанет показывать игнорируемые файлы. Если хочется всё-таки увидеть их:

```bash
git status --ignored
```

Если очень нужно добавить игнорируемый файл (бывают исключения), используйте принудительное добавление:

```bash
git add -f config.local.yml
```

**Важно.** Сам файл `.gitignore` нужно закоммитить и отправить в общий репозиторий — все, кто работает с проектом, должны игнорировать одни и те же вещи.

## Эпизод 7. Удалённые репозитории на GitHub

Локальный репозиторий хранит историю на одном компьютере. Чтобы получить резервную копию и возможность работать вместе с коллегами, его связывают с **удалённым репозиторием** — например, на GitHub.

**Шаг 1.** Зайдите на github.com, нажмите «New repository», заведите репозиторий с именем `recipes`. Важно: оставьте его **пустым** — без README, .gitignore и лицензии. Иначе при попытке связать с локальным возникнет конфликт.

**Шаг 2.** Свяжите локальный репозиторий с удалённым. GitHub после создания покажет нужный URL — например, `git@github.com:alflin/recipes.git` (где `alflin` — это имя пользователя Alfredo Linguini):

```bash
git remote add origin git@github.com:alflin/recipes.git
git remote -v        # проверить
```

`origin` — это просто стандартный алиас для основного удалённого репозитория.

**Шаг 3. Настройка SSH-ключей.** Чтобы не вводить пароль при каждом push, используют ключевую аутентификацию:

```bash
ls -al ~/.ssh                                 # есть ли уже ключи
ssh-keygen -t ed25519 -C "anna@example.com"   # создать новую пару
cat ~/.ssh/id_ed25519.pub                     # вывести публичный ключ
```

Содержимое публичного ключа скопируйте в GitHub: **Settings → SSH and GPG keys → New SSH key**. После этого проверьте подключение:

```bash
ssh -T git@github.com
```

**Шаг 4. Отправка коммитов.**

```bash
git push origin main
```

Эта команда отправляет все локальные коммиты ветки `main` в удалённый репозиторий `origin`.

**Шаг 5. Получение чужих изменений** (актуально при совместной работе):

```bash
git pull origin main
```

## Эпизод 8. Совместная работа

Alfredo пригласил коллегу — другого шефа, который тоже хочет дополнить кулинарную книгу.

**Шаг 1. Дать доступ.** На GitHub в настройках репозитория: **Settings → Collaborators → Add people**, добавить по имени пользователя.

**Шаг 2. Коллега клонирует репозиторий** к себе:

```bash
git clone git@github.com:alflin/recipes.git ~/Desktop/alflin-recipes
cd ~/Desktop/alflin-recipes
```

`git clone` сам скачивает всю историю и автоматически настраивает `origin`, указывающий на репозиторий Alfredo.

**Шаг 3. Коллега работает локально.** Создаёт новый файл `hummus.md` со своим рецептом, делает коммит и публикует:

```bash
echo "# Хумус" > hummus.md
git add hummus.md
git commit -m "Добавить рецепт хумуса"
git push origin main
```

**Шаг 4. Alfredo забирает изменения** к себе:

```bash
git pull origin main
ls            # появился hummus.md
git log       # видны коммиты коллеги
```

**Рабочий цикл.** Привычка, которая экономит много нервов: перед началом работы всегда делайте `git pull`, чтобы взять свежие изменения. Заканчиваете работу — `git add` → `git commit` → `git push`. Лучше делать **много маленьких коммитов**, чем один большой в конце дня: их проще ревьюить и откатывать по отдельности.

## Эпизод 9. Конфликты слияния

Допустим, оба шефа одновременно решили улучшить `guacamole.md` — и оба отредактировали одну и ту же строку «приготовление авокадо», но по-разному.

**Сначала коммитит коллега.** Он пишет в файл строку `* положить один авокадо в миску`, делает `git commit`, `git push origin main`. Всё проходит гладко.

**Потом коммитит Alfredo.** В его локальной копии та же строка теперь звучит иначе: `* очистить авокадо`. Alfredo делает `git commit` и `git push origin main` — и получает отказ. Git говорит: на сервере уже есть коммит, которого у тебя нет, сначала забери его.

**Alfredo делает `git pull origin main`** — и Git сообщает о **конфликте слияния**. Заглянув в `guacamole.md`, Alfredo видит специальные маркеры:

```
<<<<<<< HEAD
* очистить авокадо
=======
* положить один авокадо в миску
>>>>>>> dabb4c8c450e8475aee9b14b4383acc99f42af1d
```

Что это значит:

- Между `<<<<<<< HEAD` и `=======` — версия Alfredo (его локальный HEAD).
- Между `=======` и `>>>>>>> <hash>` — версия из коммита, который пришёл с сервера.

**Разрешение конфликта вручную.** Alfredo открывает файл, решает, что обе версии полезны, и заменяет конфликтный блок на одну строку:

```
* очистить авокадо и положить в миску
```

Затем удаляет все маркеры (`<<<<<<<`, `=======`, `>>>>>>>`), сохраняет файл и завершает слияние:

```bash
git add guacamole.md
git commit              # Git сам предложит сообщение «Merge branch 'main'»
git push origin main
```

**Главная мысль:** контроль версий не позволяет одному человеку молча затереть работу другого. Вместо этого он явно указывает на конфликт и **обязывает** разрешить его осознанно. Это раздражает в моменте, но спасает от потери чужих правок.

**Как уменьшить количество конфликтов:**

- Делать `git pull` перед началом работы и перед каждым `git push`.
- Дробить файлы на логические части — два человека реже редактируют один и тот же абзац, если файлов много.
- Договариваться, кто над каким куском работает в конкретный день.
- Использовать ветки для параллельных задач — отдельная тема за пределами этого модуля.
