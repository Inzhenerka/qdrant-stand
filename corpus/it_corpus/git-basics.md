# Основы Git

## Зачем этот курс

Git — система контроля версий, на которой построено большинство современных программистских проектов. С её помощью хранят историю изменений, откатывают неудачные правки, ведут код в команде, выкладывают работу на GitHub. Без Git не обходится ни одна стажировка и почти ни одна вакансия в разработке. Этот модуль даёт необходимый минимум для уверенной повседневной работы.

К концу модуля вы научитесь:

- заводить локальный репозиторий и фиксировать в нём изменения,
- читать историю проекта и сравнивать версии,
- откатывать неудачные правки,
- настраивать `.gitignore`, чтобы не коммитить лишнее,
- связывать локальный репозиторий с GitHub по SSH,
- работать в репозитории вместе с другим человеком,
- разрешать конфликты слияния, когда два человека правили одно и то же.

Никаких предварительных знаний Git не нужно. Желательно уверенно открывать терминал — это разбирается в модуле про Unix Shell, но базовых `cd`, `ls`, `mkdir` для начала хватит.

## Подготовка к работе

**Установка Git:**

- **Windows:** скачайте установщик с git-scm.com. Он предложит десяток опций — на все можно соглашаться по умолчанию. В комплекте идёт Git Bash — терминал, в котором примеры из модуля работают так же, как на macOS и Linux.
- **macOS:** Git обычно уже установлен. Проверьте командой `git --version`. Если нет — система сама предложит поставить Command Line Tools.
- **Linux:** `sudo apt install git` (Debian/Ubuntu), `sudo dnf install git` (Fedora), `sudo pacman -S git` (Arch).

Проверка установки:

```bash
git --version
# git version 2.43.0 — версия может отличаться, главное — что команда находится
```

Все примеры в модуле работают начиная с Git 2.23.

## Эпизод 1. Зачем нужен автоматический контроль версий

Представьте, что вы пишете большой документ — курсовую, статью, дипломный проект. После каждой правки вы сохраняете новую копию, и через месяц рядом лежат файлы вроде `kursovaya_v3_FINAL_правки_иванова_FINAL2.docx`. Какая из них самая свежая? Какая правка от какого числа? Что было в той версии, которую вы согласовали с научным руководителем неделю назад?

Системы контроля версий (Version Control Systems, VCS) решают именно эту проблему. По сути это неограниченная отмена: можно вернуться к любой предыдущей версии файла, посмотреть, что и когда было изменено, и кто это менял. Файл при этом остаётся один — все версии хранятся в специальной служебной структуре.

Современные VCS работают так: исходный файл и **набор изменений** хранятся отдельно. Систему можно попросить проиграть нужную последовательность изменений на исходник и получить любую промежуточную версию.

Такая модель естественно решает и **совместную работу**. Двое могут одновременно редактировать разные участки одного файла; система склеит их правки автоматически. Если же оба изменили одну и ту же строку по-разному — возникает **конфликт**, и система сообщит об этом, не давая никому случайно затереть чужую работу.

Git — самая распространённая VCS в мире. Это **распределённая** система: каждый участник имеет полную копию истории на своём компьютере, центральный сервер не обязателен. На практике центральный сервер всё же используется (GitHub, GitLab) — он удобен как точка обмена и резервная копия.

## Эпизод 2. Настройка Git

Перед первым использованием Git нужно представиться: указать имя и e-mail, которые будут привязаны к каждому коммиту:

```bash
git config --global user.name "Anna Petrova"
git config --global user.email "anna@example.com"
```

Имя обычно пишут латиницей — кириллица в коммитах поддерживается, но иногда вызывает проблемы в чужих инструментах. Флаг `--global` означает «для всех репозиториев на этом компьютере». Без него настройки применяются только к текущему репозиторию.

E-mail должен совпадать с тем, что указан в вашей учётной записи на GitHub (или другом хостинге) — иначе ваши коммиты не будут связываться с вашим профилем, и зелёные квадратики в графике активности не появятся.

**Окончания строк.** На Windows строки исторически заканчиваются `\r\n`, на Linux и macOS — `\n`. Чтобы Git не ломал их при переносе кода между системами, настраивают `core.autocrlf`:

```bash
git config --global core.autocrlf input    # Linux / macOS
git config --global core.autocrlf true     # Windows
```

**Текстовый редактор для сообщений коммитов.** По умолчанию Git открывает Vim, что для новичка может стать неприятным сюрпризом (выйти оттуда — задача со звёздочкой). Можно сразу выбрать привычный редактор:

```bash
git config --global core.editor "nano -w"
git config --global core.editor "code --wait"   # VS Code
```

**Имя главной ветки.** Раньше по умолчанию называлась `master`, сейчас принято `main`:

```bash
git config --global init.defaultBranch main
```

Эти команды выполняются **один раз на компьютер**. Посмотреть текущие настройки:

```bash
git config --list --global
git config --global --edit       # открыть в редакторе
```

## Эпизод 3. Создаём репозиторий

Дальше курс пойдёт по сквозному примеру. Шеф-повар Alfredo Linguini собирает свою цифровую кулинарную книгу. Он работает в папке `recipes/` и хочет отслеживать каждую правку каждого рецепта.

Создаём папку и переходим в неё:

```bash
mkdir recipes
cd recipes
```

Превращаем её в Git-репозиторий:

```bash
git init
```

Эта команда создаёт скрытую служебную папку `.git`, в которой Git будет хранить всю историю проекта. Удалите эту папку — и история исчезнет, останутся только текущие файлы.

Проверим, что репозиторий пуст и готов к работе:

```bash
git status
```

Git ответит, что коммитов ещё нет и нет файлов для отслеживания.

**Важное предупреждение.** Никогда не создавайте Git-репозиторий внутри другого Git-репозитория. Внешний попытается отслеживать содержимое внутреннего, и оба будут друг другу мешать. Если случайно так получилось — удалите вложенную папку `.git` (только её, не данные!).

## Эпизод 4. Отслеживаем изменения

Alfredo создаёт первый рецепт — файл `guacamole.md`. Положим в него заголовок и небольшой текст:

```markdown
# Гуакамоле

Простая мексиканская закуска из авокадо.
```

Сохраняем, проверяем статус:

```bash
git status
```

Git покажет файл `guacamole.md` как **untracked** — он есть в папке, но Git его пока не отслеживает. Чтобы файл попал в историю, нужны два шага:

1. **`git add`** — положить изменения в так называемую *staging area* (промежуточную область, индекс).
2. **`git commit`** — зафиксировать содержимое индекса как новую версию.

Полезная аналогия: представьте, что делаете групповое фото. `git add` — это собрать нужных людей перед камерой. `git commit` — нажать кнопку и сделать снимок. Можно собирать группу постепенно (несколько `git add` подряд), а потом одним коммитом зафиксировать всех вместе.

```bash
git add guacamole.md
git status              # теперь файл "to be committed"
git commit -m "Добавить рецепт гуакамоле"
```

Флаг `-m` задаёт сообщение коммита прямо в команде. Без него Git откроет настроенный текстовый редактор.

После коммита `git status` покажет «working tree clean» — все изменения зафиксированы.

**Правила хорошего сообщения коммита:**

- Первая строка — короткая (до 50 символов), в повелительном наклонении: «Добавить…», «Исправить…», «Обновить…».
- При необходимости — пустая строка и подробности ниже.
- Описывайте **зачем** сделано изменение, не только **что**.

Теперь Alfredo вносит правку в `guacamole.md` — добавляет ингредиенты. Прежде чем коммитить, он хочет убедиться, что именно изменилось:

```bash
git diff
```

`git diff` показывает разницу между рабочей копией и последней зафиксированной версией. Строки, помеченные `+`, добавлены; помеченные `-` — удалены.

Цикл повторяется: правка → `git add` → (опц. `git diff --staged`) → `git commit -m "…"`. На этой связке держится повседневная работа с Git.

## Эпизод 5. Изучаем историю

Просмотр всех коммитов:

```bash
git log
git log --oneline                       # короткая форма, по строке на коммит
git log --oneline --graph --all         # с графом веток
```

`--graph --all` показывает историю в виде дерева, где видно ветвления и слияния — для текущего одноветочного проекта дерево будет прямой палкой, но привычку использовать этот флаг лучше выработать сразу.

Каждый коммит идентифицируется **хешем** — уникальной строкой из 40 символов в шестнадцатеричной форме, например `f22b25e3233b4645dabd0d81e651fe074bd8e73b`. На практике достаточно первых 7 символов: `f22b25e`. Хеши неизменны: даже если вы измените одну букву в сообщении коммита, хеш станет другим — это как отпечаток пальцев для коммита.

Также есть относительные ссылки:

- `HEAD` — самый последний коммит в текущей ветке.
- `HEAD~1` — предпоследний.
- `HEAD~2` — на два коммита назад. И так далее.

Сравнить рабочую копию с разными состояниями:

```bash
git diff HEAD               # с последним коммитом
git diff HEAD~1             # с предпоследним
git diff f22b25e            # с конкретным коммитом по хешу
git show HEAD~2             # текст и сообщение коммита
```

**Откат изменений.** Если правка в рабочей копии оказалась неудачной, её можно отменить:

```bash
git restore guacamole.md             # вернуть последнюю закоммиченную версию
git restore -s f22b25e guacamole.md  # вернуть состояние из конкретного коммита
git restore --staged guacamole.md    # снять файл со staging area
```

Если файл уже был добавлен в индекс (`git add`), но ещё не закоммичен, и вы хотите отменить и индексирование, и сами правки — два шага: сначала `git restore --staged`, потом `git restore`.

**Полезный приём.** Разделяйте файлы по логическим частям (введение отдельно, заключение отдельно). Это резко упрощает откат: вы вернёте только то, что испортили, не трогая остальное.

## Эпизод 6. Игнорируем лишнее

В проектах часто появляются файлы, которые отслеживать не нужно: временные файлы редактора, кэш, скомпилированные версии, личные заметки, секреты. Чтобы Git перестал на них реагировать, создайте в корне репозитория файл `.gitignore` и перечислите шаблоны:

```
*.tmp
results/
**/*.csv
.env
notes_личное.md
```

Что означают эти строки:

- `*.tmp` — все файлы с расширением `.tmp` в любой папке.
- `results/` — вся папка `results` со всем содержимым.
- `**/*.csv` — все `.csv`-файлы на любом уровне вложенности.
- `.env` — конкретный файл `.env` (часто хранит пароли и API-ключи — его утечка в репозиторий это классическая ошибка).

Восклицательный знак отменяет игнорирование для конкретного файла:

```
*.png
!logo.png      # все PNG игнорируются, но logo.png отслеживается
```

После создания `.gitignore` команда `git status` перестанет показывать игнорируемые файлы. Если хочется всё-таки увидеть их:

```bash
git status --ignored
```

Если очень нужно добавить игнорируемый файл (бывают исключения), используйте принудительное добавление:

```bash
git add -f config.local.yml
```

**Важно.** Сам файл `.gitignore` нужно закоммитить и отправить в общий репозиторий — все, кто работает с проектом, должны игнорировать одни и те же вещи.

## Эпизод 7. Удалённые репозитории на GitHub

Локальный репозиторий хранит историю на одном компьютере. Чтобы получить резервную копию и возможность работать вместе с другими, его связывают с **удалённым репозиторием** — например, на GitHub.

**Шаг 1.** Зайдите на github.com, нажмите «New repository», заведите репозиторий с именем `recipes`. Важно: оставьте его **пустым** — без README, .gitignore и лицензии. Причина: если репозиторий уже содержит коммиты, при попытке отправить туда свою историю Git откажется — две независимые истории не склеиваются автоматически. Эту проблему можно обойти, но для первого раза проще создать пустой.

**Шаг 2.** Свяжите локальный репозиторий с удалённым. GitHub после создания покажет нужный URL — например, `git@github.com:alflin/recipes.git` (где `alflin` — это имя пользователя Alfredo Linguini):

```bash
git remote add origin git@github.com:alflin/recipes.git
git remote -v        # проверить, что remote добавлен
```

`origin` — это просто стандартный алиас для основного удалённого репозитория. Можно было бы назвать иначе, но `origin` — общепринятое соглашение.

**Шаг 3. Настройка SSH-ключей.** GitHub принимает два способа авторизации: SSH-ключи и HTTPS + персональный токен. SSH удобнее в долгой перспективе — настроили один раз, и больше пароли не вводятся:

```bash
ls -al ~/.ssh                                 # есть ли уже ключи
ssh-keygen -t ed25519 -C "anna@example.com"   # создать новую пару
cat ~/.ssh/id_ed25519.pub                     # вывести публичный ключ
```

Содержимое публичного ключа скопируйте в GitHub: **Settings → SSH and GPG keys → New SSH key**. После этого проверьте подключение:

```bash
ssh -T git@github.com
# Hi alflin! You've successfully authenticated...
```

**Шаг 4. Отправка коммитов.**

```bash
git push origin main
```

Эта команда отправляет все локальные коммиты ветки `main` в удалённый репозиторий `origin`. После первой отправки можно настроить отслеживание ветки — тогда дальше достаточно просто `git push`:

```bash
git push -u origin main      # -u запоминает связку local main <-> origin main
```

**Шаг 5. Получение чужих изменений** (актуально при совместной работе):

```bash
git pull origin main
```

## Эпизод 8. Совместная работа

Alfredo пригласил коллегу — другого шефа, который тоже хочет дополнить кулинарную книгу.

**Шаг 1. Дать доступ.** На GitHub в настройках репозитория: **Settings → Collaborators → Add people**, добавить по имени пользователя.

**Шаг 2. Коллега клонирует репозиторий** к себе:

```bash
git clone git@github.com:alflin/recipes.git ~/Desktop/alflin-recipes
cd ~/Desktop/alflin-recipes
```

`git clone` сам скачивает всю историю и автоматически настраивает `origin`, указывающий на репозиторий Alfredo. Это разовая операция — повторять при каждом запуске не нужно.

**Шаг 3. Коллега работает локально.** Создаёт новый файл `hummus.md` со своим рецептом, делает коммит и публикует:

```bash
echo "# Хумус" > hummus.md
git add hummus.md
git commit -m "Добавить рецепт хумуса"
git push origin main
```

**Шаг 4. Alfredo забирает изменения** к себе:

```bash
git pull origin main
ls            # появился hummus.md
git log       # видны коммиты коллеги
```

**Рабочий цикл.** Перед началом работы — `git pull`, чтобы взять свежие изменения коллег. По ходу работы — `git add` → `git commit` → `git push`. Делайте **много маленьких коммитов** вместо одного большого в конце дня: их проще ревьюить, откатывать по отдельности и читать в истории.

## Эпизод 9. Конфликты слияния

Допустим, оба шефа одновременно решили улучшить `guacamole.md` — и оба отредактировали одну и ту же строку «приготовление авокадо», но по-разному.

**Сначала коммитит коллега.** Он пишет в файл строку `* положить один авокадо в миску`, делает `git commit`, `git push origin main`. Всё проходит гладко.

**Потом коммитит Alfredo.** В его локальной копии та же строка теперь звучит иначе: `* очистить авокадо`. Alfredo делает `git commit` и `git push origin main` — и получает отказ. Git говорит: на сервере уже есть коммит, которого у тебя нет, сначала забери его.

**Alfredo делает `git pull origin main`** — и Git сообщает о **конфликте слияния**. Заглянув в `guacamole.md`, Alfredo видит специальные маркеры:

```
<<<<<<< HEAD
* очистить авокадо
=======
* положить один авокадо в миску
>>>>>>> dabb4c8c450e8475aee9b14b4383acc99f42af1d
```

Что это значит:

- Между `<<<<<<< HEAD` и `=======` — версия Alfredo (его локальный HEAD).
- Между `=======` и `>>>>>>> <hash>` — версия из коммита, который пришёл с сервера.

**Разрешение конфликта вручную.** Alfredo открывает файл, решает, что обе версии полезны, и заменяет конфликтный блок на одну строку:

```
* очистить авокадо и положить в миску
```

Затем удаляет все маркеры (`<<<<<<<`, `=======`, `>>>>>>>`), сохраняет файл и завершает слияние:

```bash
git add guacamole.md
git commit              # Git сам предложит сообщение «Merge branch 'main'»
git push origin main
```

**Аварийный выход.** Если что-то пошло не так и проще начать заново, можно откатить начатое слияние:

```bash
git merge --abort
```

После этого репозиторий выглядит так, будто `git pull` никогда не вызывался — все ваши локальные изменения на месте, никакого слияния не начато.

Главная роль контроля версий в этой ситуации — не позволить одному человеку молча затереть работу другого. Вместо этого Git явно указывает на конфликт и обязывает разрешить его осознанно. Это раздражает в моменте, но спасает от потери чужих правок.

**Как уменьшить количество конфликтов:**

- Делать `git pull` перед началом работы и перед каждым `git push`.
- Дробить файлы на логические части — два человека реже редактируют один и тот же абзац, если файлов много.
- Договариваться, кто над каким куском работает в конкретный день.
- Использовать ветки для параллельных задач — отдельная тема за пределами этого модуля.

- откатывать неудачные правки через `git restore`,
- исключать лишние файлы через `.gitignore`,
- связывать локальный репозиторий с GitHub по SSH и обмениваться коммитами через `git push` / `git pull`,
- клонировать чужой репозиторий через `git clone` и работать в нём вместе с другими,
- разрешать конфликты слияния — и аварийно откатывать слияние через `git merge --abort`.

Этого набора достаточно, чтобы вести любой учебный или прикладной проект — в одиночку или в команде. Следующий естественный шаг — **ветки** (`git branch`, `git checkout`, `git merge`): они нужны, чтобы вести несколько задач параллельно без помех друг другу. Их разбирают в продвинутых модулях по Git.

В следующих модулях курса вы освоите работу в терминале через Unix Shell, основы анализа данных с pandas, автоматическое тестирование с pytest и язык запросов SQL.
